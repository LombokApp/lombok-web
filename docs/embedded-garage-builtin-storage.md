# Embedded Garage as the builtin S3 storage

This document describes how Lombok embeds a [Garage](https://garagehq.deuxfleurs.fr/)
S3 daemon into every build and exposes it as the instance's **builtin storage** —
both the platform's own server storage (app bundles, icons, avatars) and an
always-available builtin storage target users can pick when creating folders.

## Background

Lombok stores files in S3-compatible backends. The platform distinguishes:

- **Server storage** — the platform's _own_ storage: app bundle uploads,
  server/app icons, avatars, dicebear styles. Read via
  `StorageProvisionService.getServerStorage()` and consumed in `app.service.ts`,
  `server-icon.service.ts`, `user-avatar.service.ts`, `folder-icon.service.ts`. It
  is **always** the embedded S3 service (a dedicated `server-storage` bucket) —
  never a stored setting.
- **Storage targets** — where a folder stores its content/metadata. When creating
  a folder the user picks one of:
  - the always-available **builtin** target (the embedded S3 service, dedicated
    `provisions` bucket), selected with `{ builtin: true }`. It is **not** a listed
    provision and has no id; the folder's location columns are stored as **NULL**
    (NULL = builtin) and the target is resolved in memory at read time.
  - an admin-defined **External storage provision**, stored in the
    `external_storage_provisions` table and selected by its id. `folder.service.ts`
    `buildLocation` copies its credentials into a `storage_locations` row.
  - a raw custom S3 location, or an override of one of the user's existing
    locations (both also persisted as `storage_locations` rows).

The earlier model stored both server storage and provisions as JSON rows in
`server_settings` (`SERVER_STORAGE` / `STORAGE_PROVISIONS`), populated only by the
dev seed from `DEV_S3_*` env. Those settings and the **fixed** `DEV_S3_*` /
`TEST_S3_*` credentials are **gone** — the embedded Garage daemon is now the single
S3 backend in every environment (dev, test, prod) with auto-generated credentials.
(The `TEST_S3_*` identifiers survive in `test.util.ts` only as thin aliases over
the auto-generated `EMBEDDED_S3_*` values.)

## Goal

Make an embedded Garage daemon a first-class, always-present part of every
Lombok build, reachable by browsers at `s3.<platform_host>`, and exposed to the
platform as:

1. the **builtin server storage**, and
2. an **always-available builtin storage target** users select with
   `{ builtin: true }` when creating a folder (not a listed provision, no id).

## Design decisions

- **Always embedded** in every build (dev, test, release, standalone) — no
  enable/disable flag.
- **Reverse-proxied through nginx in every environment.** Garage's S3 API binds
  loopback (`127.0.0.1:9000`) everywhere and is never published to the host;
  browsers always reach it via `s3.<platform_host>` through nginx. Dev/e2e used
  to expose `:9000` directly — that parity gap is now closed (see
  [Dev/test reachability](#dev--test-reachability-reverse-proxy)).
- **Single backend, every environment.** The embedded Garage daemon is the only
  S3 backend in dev, test and production — no fixed dev/test credentials, no
  DB-seeded storage rows. **No config is necessary anywhere; everything is
  generated or derived.**
- **Virtual / config-backed builtin.** The builtin server storage and target are
  _derived at runtime_ from the embedded credentials, never written to a settings
  row or the provisions table. The builtin is not a listed provision and has no id;
  a folder that uses it stores **NULL** location columns and is resolved in memory
  (`resolveFolderLocation` in `folder.service.ts`). Admin-defined **External**
  provisions are the only rows in `external_storage_provisions`.
- **DI-injected system bucket names.** Three fixed-name system buckets —
  `server-storage`, `provisions`, `uploads` — come from `coreConfig.s3SystemBuckets`
  (overridable via `S3_*_BUCKET` env). The e2e harness overrides the coreConfig
  provider to namespace them per suite.
- **Credentials auto-generated by Garage** on first boot and persisted to the
  Garage data volume; the API/seed/harness read them back via the entrypoint
  (`EMBEDDED_S3_*`). No secrets are baked into the image.
- **Garage data dirs are mapped to docker host paths** so meta/data/credentials
  survive container recreation.

## Topology: who reaches Garage, and how

The key insight is that **presigning is pure local computation**
(`aws4.sign` in `packages/api/src/storage/s3.utils.ts`) — it never contacts S3.
A presigned URL only needs the correct public `endpoint` and a matching SigV4
host. This splits traffic into two independent paths:

```
                         ┌─────────────────────────────────────────┐
  Browser  ──PUT/GET──▶  │ nginx  server_name s3.<platform_host>    │
  (presigned                proxy_pass http://127.0.0.1:9000        │
   s3.<host> URL)         └─────────────────────────────────────────┘
                                            │
                                            ▼
                                   ┌──────────────────┐
  API (server-side)  ──────────▶  │  Garage :9000     │
  rewritten to http://127.0.0.1:9000  (loopback)      │
                                   └──────────────────┘
```

- **Browser-facing** presigned URLs (folder content, app storage signed URLs)
  use the public `endpoint = <proto>://s3.<platform_host>[:port]`. nginx routes
  that host to the local Garage. The browser signs against `s3.<host>` — including
  the port on ported deployments — so nginx forwards the verbatim header
  (`proxy_set_header Host $http_host`, **not** `$host`, which strips the port) so
  the SigV4 signature still matches at Garage.
- **Server-side** S3 calls from the API (the SDK `.send()` operations and the
  presign-then-fetch `HEAD`/`GET`/`DELETE` trio in `s3.service.ts`) are rewritten
  to loopback `http://127.0.0.1:9000`. This is topology-independent: it works
  behind fronted TLS, on arbitrary public ports, and without DNS for
  `s3.<host>` inside the container. Because presign is local, the API can presign
  _against the loopback host_ and then fetch that same loopback URL — the
  signature is internally consistent.

## Implementation

### 1. Dockerfile — embed Garage in all stages

`docker/app.Dockerfile`. The Garage binary + config copy and data-dir setup were
moved out of the `dev` stage into the shared `base` stage, so every downstream
stage (dev, test, install, release, standalone-release) inherits it:

```dockerfile
COPY --from=garage /garage /usr/local/bin/garage
COPY packages/api/cmd/garage.toml /etc/garage.toml
RUN mkdir -p /var/lib/garage/meta /var/lib/garage/data && \
  chown -R bun:bun /var/lib/garage
```

`garage.toml` is single-node with loopback RPC at `127.0.0.1:3901` and
`s3_region = "auto"`. RPC is loopback-only, so the static `rpc_secret` is
acceptable for a single embedded node in every environment (including
production). The S3 API bind address is templated (`{{GARAGE_API_BIND_ADDR}}`,
substituted by `garage-provision.sh`) and now defaults to **`127.0.0.1:9000` in
every environment** — all access goes through nginx, closing off direct
nginx-bypassing reach to the shared-key S3 API on multi-interface/pod networks.
The template hook remains so an operator can widen the bind if ever needed, but
dev and e2e no longer override it (they previously bound `[::]:9000` and exposed
the port to the host).

### 2. Provisioning — auto-generate + persist credentials

`packages/api/cmd/garage-provision.sh` (shared by dev, test, and production
entrypoints) **auto-generates** the key in every environment (a fixed key can
still be imported by supplying `GARAGE_S3_ACCESS_KEY_ID` + `GARAGE_S3_SECRET_KEY`,
but no environment does so now). On first boot it mints the key — calling `garage
key create` only when no same-named key already exists (a prior boot may have
minted one but crashed before persisting; reusing it by name avoids orphaning a
second key) — reads the id + secret back via `garage key info --show-secret`, and
persists them to
`/var/lib/garage/.lombok-builtin-key` as `EMBEDDED_S3_ACCESS_KEY_ID=…` /
`EMBEDDED_S3_SECRET_ACCESS_KEY=…`. On later boots it sources the existing file
(idempotent, guarded by file existence). The parsed id/secret are asserted
non-empty before the file is written (and again after sourcing it) — a failed
scrape aborts loudly rather than persisting an empty key file that would poison
every later boot.

It then creates the **three system buckets** (`server-storage`, `provisions`,
`uploads`, names overridable via `GARAGE_*_BUCKET`) plus any `GARAGE_EXTRA_BUCKETS`,
grants the key full access on all of them, and applies CORS (`garage-set-cors.ts`)
to the **browser-facing** ones (`provisions`, `uploads`, extras) — `server-storage`
is server-side only, so it gets no CORS rule. (Garage, unlike MinIO, is not
permissive by default, so browser-direct presigned uploads/downloads need an
explicit rule.) The `uploads` bucket is reserved for presigned **staged uploads**;
that feature lands on a separate branch, so here the bucket is provisioned (and
CORS'd) ahead of it but not yet consumed. The e2e key is also granted
`createBucket` so the harness can mint per-suite namespaced buckets over S3.

### 3. Entrypoint — start Garage and export credentials

`packages/api/cmd/entrypoint.sh` gained a Garage section (mirroring the existing
`EMBEDDED_POSTGRES` block, but **unconditional**) that runs before the backend
launch. It calls the provisioner with `GARAGE_S3_KEY_NAME=lombok-builtin` (no
access key → auto mode; bucket names come from coreConfig defaults), then sources
the key file and exports the credentials so the foreground `bun start` child
inherits them:

```sh
. /var/lib/garage/.lombok-builtin-key
export EMBEDDED_S3_ACCESS_KEY_ID EMBEDDED_S3_SECRET_ACCESS_KEY
export EMBEDDED_S3_REGION="auto"
```

The dev and test entrypoints do the same (auto mode + source/export the key).
The dev entrypoint additionally sets `GARAGE_EXTRA_BUCKETS=external-demo` (the seed
wires it up as an example External provision); the test entrypoint pins
`EMBEDDED_S3_ENDPOINT=http://s3.<host>:8080` because its nginx serves S3 on a
different port than the logical platform port.

### 4. nginx — route `s3.<platform_host>` to Garage

`packages/api/nginx/nginx.conf` declares a dedicated `server` block (before the
regex-matched app/tunnel blocks, so this exact host wins):

```nginx
server {
    listen 0.0.0.0:8080;
    server_name s3.{{PLATFORM_HOST}};
    client_max_body_size 0;            # large/streamed uploads
    location / {
        proxy_pass http://127.0.0.1:9000;
        proxy_set_header Host $http_host;   # verbatim host:port so browser SigV4 matches
        ...
    }
}
```

`{{PLATFORM_HOST}}` is substituted by the `sed` in `entrypoint.sh`.

### 5. API — virtual builtin server storage + target

`packages/api/src/storage/embedded-s3.ts` is the single source of truth:

- `getEmbeddedS3Config()` — memoized from env (`EMBEDDED_S3_*` +
  `PLATFORM_HOST`/`PLATFORM_HTTPS`/`PLATFORM_PORT`). Builds the public
  `endpoint = <proto>://s3.<host>[:port]` (via `buildPlatformOrigin`, or
  `EMBEDDED_S3_ENDPOINT` when pinned). Holds creds + endpoint only — **no bucket**
  (bucket names come from coreConfig DI). Present in every environment now.
- `buildEmbeddedServerStorage(bucket)` → `ServerStorageWithSecret` in its
  dedicated bucket (no prefix).
- `buildEmbeddedStorageProvision(bucket, ctx?)` → the builtin provision shape
  (`Omit<StorageProvisionWithSecret, 'id' | 'userId'>` — **no id**, no `builtin`
  flag; `provisionTypes: ['CONTENT','METADATA']`). Used only to resolve a folder's
  builtin location in memory, never listed. The optional `ctx` (`{ folderId,
  userId }`) yields the per-folder prefix.
- `rewriteToInternalEndpoint(endpoint)` → returns `http://127.0.0.1:9000` when
  `new URL(endpoint).host` equals the embedded public host, else the endpoint
  unchanged.

`StorageProvisionService` (`storage-provision.service.ts`) owns the storage model
(it injects `coreConfig` for the bucket names):

- `getServerStorage()` always returns the embedded builtin
  (`buildEmbeddedServerStorage(s3SystemBuckets.serverStorage)`, a resolved Promise,
  no DB round trip).
- `listExternalStorageProvisionsAsUser()` returns **only** the
  `external_storage_provisions` rows (no builtin prepended), filtered by
  `provisionType`.
- `getStorageProvisionById()` / `create` / `update` / `delete` operate solely on
  `external_storage_provisions`. There is no reserved builtin id and no
  builtin-mutation guard — the builtin simply isn't a provision.

Admin CRUD is served by the **`external-storage-provisions`** controller at
`/api/v1/server/external-storage-provisions`.

When a user creates a folder with the **builtin** target (`{ builtin: true }`),
`folder.service.ts` `buildLocation` returns **null** (no `storage_locations` row)
and the folder's `content_location_id` / `metadata_location_id` columns are stored
NULL. `resolveFolderLocation` returns a `FolderStorageTarget` (addressing-only)
with `kind: 'BUILTIN'` and **no `id`/`userId`**, synthesized from the embedded
creds + `provisions` bucket + a folder-derived prefix. External / custom / user
locations have a real `storage_locations` row and resolve to `kind: 'SERVER'` /
`'USER'`. `getFolder` / `getFolderAsUser` / `listFoldersAsUser` attach the resolved
targets, so the `Folder` type carries `contentLocation: FolderStorageTarget`.
Selecting the builtin by a provision id is no longer possible — that id was
removed; `{ builtin: true }` is the only way.

**Server storage is never a stored setting.** It is always the embedded service in
the `server-storage` bucket. The `SERVER_STORAGE` / `STORAGE_PROVISIONS` settings
keys and the `server-storage-input` DTO were removed entirely.

### 6. Endpoint rewrite — server-side sites only

`rewriteToInternalEndpoint(...)` is applied at exactly **four** server-side sites
in `s3.service.ts`:

1. inside `configureS3Client` (covers every `.send()` op, incl.
   `storage-location.service.ts`), and
2–4. in the three presign-then-fetch methods `s3HeadObject`,
`s3GetBucketObject`, `s3DeleteBucketObject`.

It is deliberately **not** applied in the browser-facing
`createS3PresignedUrls` callers (folder content / app storage signed URLs) —
those must keep the public `s3.<host>` host so the browser can reach them.

### 7. Types + UI

The resolved folder location is exposed as **`StorageTarget`**
(`storage-target.dto.ts`) — a discriminated union on `kind`
(`BUILTIN | SERVER | USER`); `id`/`userId` exist only on the row-backed
(`SERVER`/`USER`) variants, absent for `BUILTIN`. The `storage_locations`
discriminant column and the resolved discriminant were renamed
`provider_type` → **`kind`**, and the old `builtin` flag on the provision schema
was removed (the builtin is no longer a listed provision). In the UI, the admin
storage tab lists **External** provisions only (no builtin row or badge), and the
create-folder form always offers a static **Built-in storage** option that emits
`{ builtin: true }`. The OpenAPI spec + generated types were regenerated
(`./dx generate openapi`).

### 8. Supervision — keep Garage + Postgres alive

`entrypoint.sh` launches Garage and (embedded) Postgres as background children,
then starts a watchdog (`supervise &`) before the foreground backend. Every
`SUPERVISOR_INTERVAL_SECONDS` (default 10s) it polls `garage status` and
`pg_isready` and recovers whichever daemon is unhealthy. Both relaunches are
idempotent — data dirs, cluster layout and keys persist on disk.

The watchdog tracks each daemon's PID, so before relaunching it first stops a
**wedged-but-alive** instance (one still holding its ports) via a per-service
signal sequence — `TERM`/`INT` → grace → escalation signal → `KILL` — rather than
spawning a duplicate that fails to bind. It requires `SUPERVISOR_FAIL_THRESHOLD`
(default 2) consecutive failed checks before acting (a transient-blip guard), and
rate-limits relaunches with per-service exponential backoff capped at
`SUPERVISOR_MAX_BACKOFF_SECONDS` (default 60s); a healthy check resets both
counters. It never exits — a crash-looping dependency is retried forever at the
capped cadence rather than masked or made fatal.

### 9. Standalone compose — persist Garage data

`docker-compose.standalone.demo.yml` maps the Garage data root to a host dir
alongside the Postgres mount:

```yaml
- ./.docker/garage-standalone/:/var/lib/garage
```

One mount covers `meta`, `data`, and the persisted `.lombok-builtin-key`, so the
generated credentials and stored objects survive container recreation.

## Dev / test reachability (reverse-proxy)

Dev and e2e reach Garage the same way production does — through nginx at
`s3.<platform_host>`, never the raw `:9000` port:

- **`dev-nginx.conf`** carries the same `s3.{{PLATFORM_HOST}}` server block as the
  production `nginx.conf`.
- **`docker-compose.yml` (dev):** no `DEV_S3_*` env; the in-container API reaches
  S3 at `s3.<platform_host>:8080` via nginx (`extra_hosts` resolves the name to
  loopback), and the `:9000` host port mapping is removed.
- **`dev-entrypoint.sh` / `test-entrypoint.sh`:** Garage binds loopback; both
  auto-generate the key and source/export `EMBEDDED_S3_*`. The test entrypoint
  appends `127.0.0.1 s3.<platform_host>` to `/etc/hosts` and pins
  `EMBEDDED_S3_ENDPOINT=http://s3.<host>:8080` (its nginx S3 port differs from the
  logical `PLATFORM_PORT`).
- **Core-worker sandbox (`run-worker.ts`):** the nsjail jail bind-mounts
  `/etc/hosts` (read-only) alongside `/etc/resolv.conf` so the sandbox can resolve
  the `s3.<host>` endpoint (real DNS in prod, `/etc/hosts` loopback in dev/e2e).

## Dev / test vs production

The storage **model** is now identical across environments — only credentials and
routing detail differ:

| | dev / test | production (release / standalone) |
| --- | --- | --- |
| Garage embedded | yes | yes |
| Credentials | auto-generated, persisted | auto-generated, persisted |
| `EMBEDDED_S3_*` env | set by entrypoint | set by entrypoint |
| Server storage / targets | **builtin** (config-backed) + External provisions table | same |
| System buckets | namespaced per e2e suite (dev: fixed names) | fixed names |
| Garage S3 bind | loopback `127.0.0.1:9000` | loopback `127.0.0.1:9000` |
| S3 reachability | browser/API via nginx `s3.<host>:8080` | browser via nginx `s3.<host>`; server-side loopback rewrite |

The e2e harness namespaces the three system buckets per suite and overrides the
`coreConfig` provider so each suite's server storage and builtin target resolve to
its own buckets.

## Out of scope / follow-ups

- **One embedded key, internal-only.** A single full-owner key backs all three
  system buckets. This is acceptable because the key never leaves the API process —
  every browser/app/worker boundary crosses a **presigned URL**, never the raw
  credential — so splitting it per bucket buys nothing (whatever can read one key
  can read them all). The property to preserve is presign-only exposure. Rotation
  of **External** provision keys is supported (`rotateAccessKeyAsAdmin` updates
  `external_storage_provisions`); the embedded key has no rotation flow yet (builtin
  folders have NULL locations, so they're naturally excluded from the user-key
  rotation path). A graceful embedded-key rotation — new key, grant, swap, delete
  the old key only after the max presigned-URL TTL — is a follow-up.
- Admin-defined **non-default server storages** (a routing layer on top of the
  builtin).
- Per-install Garage `rpc_secret` generation (current loopback-only RPC makes
  this optional).

## Verification

1. **Build:** `./dx release standalone <ver>`; confirm the image contains
   `/usr/local/bin/garage` and `/etc/garage.toml`.
2. **Boot:** run via `docker-compose.standalone.demo.yml`; entrypoint logs show
   the daemon up, layout applied, key file created, the `server-storage` /
   `provisions` / `uploads` buckets created, CORS applied to the browser-facing
   ones.
3. **Routing:** `curl -i http://s3.<platform_host>:<port>/` reaches Garage
   (S3 XML/403, not nginx 404).
4. **External provisions:** as admin, `GET /api/v1/server/external-storage-provisions`
   returns only admin-defined External rows (the builtin is **not** listed);
   `POST`/`PUT`/`DELETE` manage those rows.
5. **Builtin folder:** create a folder with **Built-in storage** (`{ builtin: true }`)
   — its `content_location_id` / `metadata_location_id` are NULL and the response
   `contentLocation.kind` is `BUILTIN` (no `id`); upload a file (browser →
   presigned `s3.<host>` → `provisions` bucket) and read it back. Create one against
   the seeded **External** provision → a `storage_locations` row with
   `kind = 'SERVER'` is written. Confirm server icon / avatar / app install work via
   the `server-storage` bucket.
6. **Persistence:** restart the container; the same key file is reused, objects
   survive, and the host `./.docker/garage-standalone/` holds `meta`/`data`.
7. **Dev routing:** in the dev container, confirm `:9000` is not published, and
   that browser uploads + worker analyze both succeed via `s3.<host>:8080`
   through nginx.
8. **Regression:** `./dx check all` is green. `./dx e2e api` needs a clean run
   after the e2e image is rebuilt with the standard method (the entrypoint +
   bucket-provisioning changes are baked into the image).
